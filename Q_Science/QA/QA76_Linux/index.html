  • Titus, Chris: How To Secure A Server, Chris Titus Tech, 11.10.2019. See also How to Secure A Web Server (10.10.2019).
    The video shows how to secure an Ubuntu webserver.
    1. The first step is enabling ufw on the server. sudo ufw limit 22/tcp limits the number of connections from a specific IP address to the standard port for SSH. sudo ufw allow 80/tcp makes sure that HTTP connections will be allowed once the firewall is enabled. sudo ufw allow 443/tcp does the same for HTTP connections over TLS/SSL.
    2. sudo ufw enable then enables the firewall. sudo ufw status can be used to verify the settings.
    3. The next step is setting up the default policies. sudo ufw default deny incoming denies all incoming traffic. sudo ufw default allow outgoing allows outgoing traffic. sudo ufw reload makes sure that these policies become effective.
    4. On the client PC, create an SSH key pair: ssh-keygen -t rsa. (Don't skip setting a passphrase.) The public key can be copied to the server using various method, e.g. SCP. Chris demonstrates an easy method: ssh-copy-id -i ~/.ssh/id_rsa.pub user@example.com (the file location, the user name and server name or the IP address will vary). You can then test the login, e.g. ssh user@example.com. (Also make a backup of your SSH keys.)
    5. Next, as root user, edit the file /etc/sshd_config: sudo su, then nano /etc/sshd_config and ensure that the following settings are enabled:
      ChallengeResponseAuthentication no
      PasswordAuthentication no
      UsePAM no
      PermitRootLogin no (to disable logging in as root).
      After this, restart the SSH daemon: systemctl restart sshd. Then exit the server and reconnect using the same log in as above to check that the settings have taken effect and are set correctly. Also use a different PC without the private SSH key to check that log-in attempts without a key are denied.
    6. Log back in to the server, go to ~/.ssh and check the list of authorised keys. Chris then demonstrates how to add a public SSH key generated on a different PC.
    7. On the server, open the file /etc/sysctl.conf: sudo nano /etc/sysctl.conf. In this file, you need to uncomment lines to enable spoof protection, disable ping (the ICMP protocol) and reject IP source route packets. Then either reboot the server or run sysctl -p to load these settings into the kernel.
    8. The next step is preventing IP spoofing. Run sudo nano /etc/host.conf; change the line order hosts, binds to order binds, hosts and change the line multi on to nospoof on.
    9. The next step is installing and setting up fail2ban: sudo apt install fail2ban sudo systemctl enable fail2ban and sudo systemctl start fail2ban. Finally, check that everything is running as expected: sudo systemctl status fail2ban.
    10. Finally, sudo netstat -tunlp tells you what services the server exposes to the world.
    This covers the basics of a security setup. In addition, it is necessary to set up monitoring (especially on a bigger server or if you are running a CMS) .

  • Let's Talk To Linux Kernel Developer Greg Kroah-Hartman | Open Source Summit, 2019 (TFiR - Open Source & Emerging Technologies on YouTube, 32 minutes, 03.09.2019). Greg Kroah-Hartman says that security vulnerabilities such as Spectre and Meltdown were the consequence of hardware developers (Intel) taking shortcuts in order to achieve speed improvements. He admits that the OpenBSD were right when they said that hyperthreading should be disabled when you can't trust the users. Researchers are now finding new vulnerabilities by reading Intel's patents The ultimate solution to the vulnerabilities is fixing the hardware; Intel was affected by these vulnerabilities, but AMD, PowerPC and ARM were not, so it is possible to design hardware differently. In the mean time, some security issues can be worked around in the BIOS, but since millions of affected devices are alrready in use, it is also necessary to fix issues in the opereating system. Many of the operating system fixes involving flushing buffers, which affects performance. In addition to these reactive security fixes, the kernel developers have also been adding proactive security features for several years. Android vendors also need to take the security fixes added by the Linux kernel developers. Greg Kroah-Hartman has also audited the kernels of sume Android phone vendors. He also talks about measures to make sure that kernal development can continue if someone like himself or Linus Torvalds were to be hit by a bus one day. He talks about the Linux distros mailing list, a private mailing list which helps (the) various Linux distributions with co-ordination related to vulnerability releases. You need to do work in order to join this mailing list; Microsoft has a Linux distribution and much of its Azure infrastructure relies on Linux. Companies contribute to Linux in a very selfish manner, but that is good because everybody has the same problems. It costs more for a company to maintain their own fork of Linux than to contribute their improvements upstream. Even Amazon contributes to the kernel now. Linux on the desktop matters to Greg Kroah-Hartman as a developer; he sees it as a tool. He talks about the different philosphy of Richard Stallman; he says GPL 2 was a great contribution to open source. He says that GPL 3 introduced restrictions on how software can be used and most of the kernel developer community did not like these restrictions. These restrictions may be appropriate for other software but they were not compatible with the direction the kernel developers had in mind. He talks about his move to the Netherlands (after living in Paris for some time); some other kernel developers also live in The Hague. He mentions that he uses Arch, but his build system runs on Fedora. He moved from Gentoo to Arch because of something he needed that was not available on Gentoo. He also likes the rolling-release system and Arch works really well. He says the systemd documentation on the Arch wiki is an amazing piece of documentation. Another advantage from a developer point of view is that Arch stays as close to upstream as possible. Fedora is also good but is six months behind on some things. He adds that possibly 80% of the world's Linux servers use either Debian or the kernel.org kernel, without Red Hat or OpenSUSE having lost market share recently.
